# HyperGraphRAG 教育知识超图体系 设计文档

> 本文档描述基于 HyperGraphRAG 底层架构，面向教育行业的多空间知识超图体系设计。
> 文档状态：**设计阶段，代码尚未实现**

---

## 一、整体设计思想

### 1.1 核心理念

- **多空间分层**：将知识按照抽象层次划分为多个空间（Layer），每个空间承载不同粒度的知识实体
- **超图结构**：以 `(g_source, r, g_target)` 三元组为基本超边结构，其中 g 是节点集合（子图），r 是映射关系集合（函数集合），超边本身不是独立实体，其内容来源于所连接节点的信息
- **分形特性**：任意子图内部仍由实体和边组成，与整体结构同构——子图可以继续作为另一个超边的端点
- **函数作为关系**：所有关系（Contains、Depends、Prerequisite 等）均以函数形式表达，如 `fun_contains(kp, kf) → Boolean`

### 1.2 空间分层总览

```
┌─────────────────────────────────────────────────────────────────┐
│                          L3 目标空间                              │
│                     LearningGoal（学习目标）                       │
│                 level: 了解(1) / 理解(2) / 掌握(3) / 运用(4)       │
└────────────────────────────┬────────────────────────────────────┘
                             │ (g₃, r₃₁, g₁)
                             │ 超边：关联目标与知识点子图
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                          L1 知识点空间                           │
│                     KnowledgePoint（知识点）                       │
│                 关系: Contains / Depends / Related                │
└────────────────────────────┬────────────────────────────────────┘
                             │ (g₁, r₁₂, g₂)
                             │ 超边：关联知识点与知识片段子图
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                          L2 知识片段空间                          │
│                  KnowledgeFragment（知识片段）                     │
│           ftype / content / corpus_id / chapter / source_file     │
└─────────────────────────────────────────────────────────────────┘
```

## 二、实体定义

### 2.1 L1 知识点（KnowledgePoint，KP）

| 属性 | 类型 | 说明 |
|------|------|------|
| `id` | string | 全局唯一标识，格式建议 `kp_<uuid>` |
| `name` | string | 知识点名称 |
| `domain` | string | 所属领域/学科，从教材目录自动推断得到 |

**设计说明**：
- `domain` 由 LLM 自动从输入教材的目录结构推断生成
- 教材的章节目录层级天然对应 domain 的层级划分
- 同一教材的不同章节自动映射为不同的 domain 值

---

### 2.2 L2 知识片段（KnowledgeFragment，KF）

| 属性 | 类型 | 说明 |
|------|------|------|
| `id` | string | 全局唯一标识，格式建议 `kf_<uuid>` |
| `kp_id` | string | 所属知识点 ID，指向 L1 中的 KP |
| `ftype` | string | 源文件类型，如 `pdf`、`txt`、`md`、`docx` 等，由用户输入文件的格式决定 |
| `content` | string | 片段正文内容，从源文件中提取 |
| `corpus_id` | string | 语料库 ID，同领域多本教材可归入同一语料库 |
| `chapter` | string | 所属章节，来源为教材章节编号或标题 |
| `source_file` | string | 源文件 ID，由用户输入文件的 ID 决定 |
| `domain` | string | 所属领域，与 KP.domain 一致 |

**ftype 示例**（由用户输入文件决定）：

| ftype 值 | 含义 |
|----------|------|
| `pdf` | PDF 文档 |
| `txt` | 纯文本文件 |
| `md` | Markdown 文件 |
| `docx` | Word 文档 |
| `html` | 网页文件 |
| ... | 由具体输入文件格式决定 |

**设计说明**：
- `corpus_id` 用于将同领域多本教材归为一组，便于跨教材检索
- `kp_id` 是连接 L1 与 L2 的桥梁，一个 KP 可对应多个 KF
- `ftype` 完全由用户输入决定，不做预设约束

---

### 2.3 L3 学习目标（LearningGoal，LG）

| 属性 | 类型 | 说明 |
|------|------|------|
| `id` | string | 全局唯一标识，格式建议 `lg_<uuid>` |
| `name` | string | 目标名称 |
| `level` | int | 认知层次，1~4 四级 |
| `description` | string | 目标详细描述 |

**level 定义**：

| level 值 | 名称 | 认知分类（参考 Bloom） |
|----------|------|----------------------|
| 1 | 了解 | 记忆、识别 |
| 2 | 理解 | 解释、举例 |
| 3 | 掌握 | 应用、实践 |
| 4 | 运用 | 分析、评价、创造 |

## 三、关系定义

### 3.1 L1 空间内部关系（KP ↔ KP）

#### 3.1.1 Contains（包含）

```
KP ──Contains──▶ KF

含义：一个知识点包含多个知识片段
类型：一对多（1 KP → N KF）
     同时允许多对多（1 KF 可从属多个 KP）
函数形式：fun_contains(kp_id, kf_id) → Boolean
```

#### 3.1.2 Depends（前置依赖）

```
KP₁ ──Depends──▶ KP₂

含义：KP₁ 的学习以 KP₂ 为前置条件
类型：有向单跳（KP → KP）
设计说明：有向边，依赖链不可环（需环检测）
函数形式：fun_depends(kp_src, kp_tgt) → Boolean
```

#### 3.1.3 Related（关联）

```
KP₁ ⇄ Related ⇄ KP₂

含义：两个知识点存在关联但无严格依赖
类型：双向等价（两节点互为 related）
设计说明：无方向性，可用于知识图谱扩展和推荐学习
函数形式：fun_related(kp_a, kp_b) → Boolean
```

---

### 3.2 L3 空间内部关系（LG ↔ LG）

#### 3.2.1 Prerequisite（前置目标）

```
LG₁ ──Prerequisite──▶ LG₂
LG ──Prerequisite──▶ KP   （跨空间）

含义：
  · LG₁→LG₂：达成 LG₁ 之前需先达成 LG₂（L3 内部，同空间）
  · LG→KP：达成该目标之前需先掌握某知识点（跨空间，L3→L1）
类型：有向（指向同空间 LG 或跨空间 KP）
空间：L3 内部（L3→L3）以及跨空间（L3→L1）
函数形式：fun_prerequisite(lg_src, tgt) → Boolean
         其中 tgt ∈ {LG, KP}
```

> **澄清**：Prerequisite 支持跨空间指向（L3→L1），即学习目标可以直接依赖某个知识点作为前置。超边 `(g₃, r₃₁, g₁)` 即承载这一跨空间映射。

---

### 3.3 跨空间关系（超边）

> **重要**：r12、r31 等超边本身是**独立可索引的实体**，不是纯链接关系。

#### 3.3.1 (g₁, r₁₂, g₂) — L1 与 L2 的链接

```
L1知识点空间  ◀─────(g₁, r₁₂, g₂)─────▶  L2知识片段空间

g₁：L1 中的 KP 节点集合（源子图）
g₂：L2 中的 KF 节点集合（目标子图）
r₁₂：映射关系集合，包含：
     - Contains(kp, kf)：包含关系

设计说明：
  - 这是一个超边，r₁₂ 是映射关系的集合（函数集合），不是有独立 content 的实体
  - 实际的 content 在 KF 节点中（KF.content），不在超边本身上
  - g₁ 和 g₂ 各自可能包含多个节点
  - 该超边体现了 L1 空间到 L2 空间的知识映射过程
  - 超边在 HyperGraphRAG 底层存储为可索引结构，但其 content 来源为所连接节点的信息
```

#### 3.3.2 (g₃, r₃₁, g₁) — L3 与 L1 的链接

```
L3目标空间  ◀─────(g₃, r₃₁, g₁)─────▶  L1知识点空间

g₃：L3 中的 LG 节点集合（源子图）
g₁：L1 中的 KP 节点集合（目标子图）
r₃₁：映射关系集合，描述目标到知识点的映射
     例如：Goal "理解函数概念" → KP "函数定义", KP "函数性质"

设计说明：
  - r₃₁ 超边将学习目标与具体知识点关联
  - 同一学习目标可对应多个知识点（g₁ 包含多节点）
  - 该超边支持"目标驱动"的知识检索场景
  - Prerequisite 允许跨空间指向：L3→L3（同空间）和 L3→L1（跨空间）
```

## 四、超图嵌套分形结构

### 4.1 分形原理

任意节点集合 g 内部仍由**实体 + 边**构成，与整体结构同构。

```
(g₁, r₁₂, g₂) 整体作为一个超边

其中 g₁ 内部：
  KP₁ ⇄Related⇄ KP₂
  KP₁ ──Depends──▶ KP₃

其中 g₂ 内部：
  KF₁ ←─Contains─ KF₂（KP₁）
  KF₃ ←─Contains─ KF₄（KP₂）
```

这种嵌套结构使得：
- **向上**：超边可以继续作为更高层超边的端点
- **向下**：任意节点可以展开为子图进行分析

### 4.2 检索粒度

| 检索粒度 | 说明 |
|---------|------|
| 节点级 | 检索单个 KP、KF 或 LG |
| 子图级 | 检索 g₁、g₂ 等节点集合 |
| 超边级 | 检索 r₁₂、r₃₁ 等映射关系及其连接的子图 |
| 路径级 | 检索 KP → … → LG 等跨空间路径 |

## 五、domain 的层级设计

### 5.1 设计思想

利用教材目录的天然层级结构，将目录转换为 domain 的层级体系。

### 5.2 转换规则

```
教材目录示例（数学）：
├── 第1章 函数
│   ├── 1.1 函数的概念
│   │   ├── 定义
│   │   ├── 例题
│   │   └── 习题
│   └── 1.2 函数的性质
│       ├── 单调性
│       └── 奇偶性
└── 第2章 导数
    └── 2.1 导数定义

LLM 推断后的 domain 层级：

level-1:  "数学"
level-2:  "函数", "导数"
level-3:  "函数.概念", "函数.性质", "导数.定义"
level-4:  "函数.概念.定义", "函数.概念.例题", "函数.性质.单调性", ...
```

### 5.3 domain 属性

- **来源**：LLM 自动从教材目录结构推断
- **格式**：`字符串`，使用 `.` 作为层级分隔符
- **粒度**：由教材目录深度决定，不做硬性限制
- **一致性**：KF 的 domain 自动继承自其 kp_id 所指向的 KP.domain

## 六、多 Agent 协作体系

### 6.1 整体架构

```
┌──────────────────────────────────────────────────┐
│                 User Interface                    │
│              （教师/学生/管理员）                   │
└────────────────────┬─────────────────────────────┘
                     │
┌────────────────────▼─────────────────────────────┐
│              Orchestrator Agent                   │
│         （全局协调者，主控 agent）                   │
│  · 理解用户意图                                   │
│  · 拆解任务分发到专业 Agent                        │
│  · 整合结果返回用户                               │
└──────┬───────────────┬───────────────┬────────────┘
       │               │               │
┌──────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
│  Indexing   │ │   Query     │ │ Evaluation  │
│   Agent     │ │   Agent     │ │   Agent     │
│ (知识构建)   │ │ (知识检索)   │ │ (学习评估)   │
└─────────────┘ └─────────────┘ └─────────────┘
```

### 6.2 各 Agent 职责

#### 6.2.1 Orchestrator Agent（主控）

- 接收用户自然语言输入
- 判断任务类型（知识构建 / 知识检索 / 学习评估）
- 负责任务分发与结果聚合
- 管理对话上下文和记忆

#### 6.2.2 Indexing Agent（知识构建）

- 接收教材文件，调用 LLM 推断目录 → domain
- 执行分块（chunking）：按 ftype 和章节切分
- 执行实体提取：
  - LLM 提取 KP（知识点）：id, name, domain
  - LLM 提取 KF（知识片段）：id, kp_id, ftype, content, corpus_id, chapter, source_file, domain
  - LLM 提取 KP 间关系：Contains, Depends, Related
- 写入超图存储（基于 HyperGraphRAG 底层）
- 输出：超图索引完成报告

#### 6.2.3 Query Agent（知识检索）

- 接收用户查询（自然语言）
- 决定检索模式：
  - `local`（L2 片段检索）：查具体知识点内容
  - `global`（L1 知识结构检索）：查知识点关联网络
  - `hybrid`（混合）：结合 L1 + L2 + L3
  - `goal-driven`（L3 目标驱动）：从学习目标反向检索
- 构建 `(g_source, r, g_target)` 超边查询
- 整合多空间检索结果生成答案

#### 6.2.4 Evaluation Agent（学习评估）

- 根据 LG（LearningGoal）和 r₃₁ 超边，评估学生对各目标的达成度
- 输入：学生答题记录 / 学习行为数据
- 输出：level（1-4）维度的掌握程度雷达图

### 6.3 Agent 间通信协议

```
Orchestrator → IndexingAgent:
  { action: "index", files: [...], corpus_id: "math_grade10" }

IndexingAgent → Orchestrator:
  { status: "done", kp_count: N, kf_count: M, domains: [...] }

Orchestrator → QueryAgent:
  { action: "query", text: "函数单调性的例题有哪些？", mode: "hybrid" }

QueryAgent → Orchestrator:
  { answer: "...", contexts: [...], confidence: 0.92 }

Orchestrator → EvaluationAgent:
  { action: "evaluate", student_id: "...", goals: ["lg_001", "lg_002"] }

EvaluationAgent → Orchestrator:
  { results: { lg_001: 3, lg_002: 2, ... }, radar_data: {...} }
```

## 八、与 HyperGraphRAG 底层存储的映射

> 以下为当前 HyperGraphRAG 代码层面对本设计各实体的存储映射关系。

### 8.1 存储命名空间

| 本设计实体 | HyperGraphRAG 存储 | 说明 |
|-----------|------------------|------|
| KP（L1 知识点） | `chunk_entity_relation_graph`（节点，role=entity） | 图节点，属性含 id/name/domain |
| KF（L2 知识片段） | `text_chunks` KV + `chunks_vdb` 向量库 | KV 存原始 content，向量库支持语义检索 |
| LG（L3 学习目标） | `chunk_entity_relation_graph`（节点，role=goal） | 新增节点类型 |
| Contains / Depends / Related | `chunk_entity_relation_graph`（边） | NetworkX 边，含关系类型及函数标记 |
| r₁₂ 超边 | 通过 KP 的 Contains 边实现 | r₁₂ 是 KP→KF 包含关系的函数集合，不独立存储 |
| r₃₁ 超边 | 通过 LG→KP 的 Prerequisite 边实现 | 同上，跨空间依赖通过有向边承载 |
| KP.embedding | `entities_vdb` | name + description 向量化 |
| KF.embedding | `chunks_vdb` | content 向量化 |
| LG.embedding | `entities_vdb`（新增 namespace） | 新增 goal 向量索引 |

### 8.2 待扩展的实体类型

当前 HyperGraphRAG 代码中节点 role 仅支持 `entity` 和 `hyperedge`，需新增：

| role | 位置 | 说明 |
|------|------|------|
| `goal` | `chunk_entity_relation_graph` 节点 | L3 学习目标节点 |
| `prerequisite` | `chunk_entity_relation_graph` 边 | 新增跨空间依赖边类型（L3→L1） |

## 九、设计决策汇总

| 编号 | 决策项 | 选择 |
|------|--------|------|
| D1 | domain 获取方式 | LLM 自动推断教材目录结构 |
| D2 | Contains 关系基数 | 一对多，同时允许多对多 |
| D3 | Prerequisite 指向 | 支持 L3→L3（同空间）和 L3→L1（跨空间）两种 |
| D4 | 超边可索引性 | r12、r31 等超边是映射关系集合（函数集合），无独立 content，通过所连接节点的内容实现检索 |
| D5 | L1↔L2 函数形式 | 三元组形式：Contains(KP, KF)，具体函数形式抽象 |
| D6 | ftype 类型开放 | 由用户输入文件格式决定，不预设枚举 |
| D7 | corpus_id 定义 | 同领域多本教材归入同一语料库 |
| D8 | LG.level 作为属性 | level 为整数 1-4，属性的独立维度 |
| D9 | 嵌套分形 | 任意子图可继续作为超边端点，支持无限嵌套 |
