# HyperGraphRAG 改造执行计划

> 本计划描述如何将现有 HyperGraphRAG 代码改造为支持多空间知识超图体系。
> 改造原则：不改变底层存储架构，只扩展节点类型、关系类型和数据字段。

---

## 一、改造涉及的存储结构

### 1.1 节点类型扩展

现状 HyperGraphRAG 的 `chunk_entity_relation_graph` 节点只有一种 role：

```
现状：role = "entity" 或 "hyperedge"
```

改造后：

```
role = "concept"    ← L1 概念（对应我们设计的 Concept）
role = "fragment"   ← L2 片段（对应我们设计的 KnowledgeFragment）
role = "goal"       ← L3 目标（对应我们设计的 LearningGoal，可选）
role = "hyperedge"  ← 保留（用于底层超边兼容）
```

### 1.2 边的类型扩展

现状只有一种边：实体之间的普通边（用于 Depends / Related / Contains）。

改造后新增：

```
Depends      ← L1 内部，概念与概念
Related      ← L1 内部，概念与概念
Contains     ← L1→L2 跨空间，概念→片段
Prerequisite ← L3→L3（同空间）或 L3→L1（跨空间）
```

### 1.3 向量索引扩展

```
entities_vdb → concept_vdb
  向量来源：concept.name + concept.domain 拼接

chunks_vdb → fragment_vdb
  向量来源：fragment.content

hyperedges_vdb → relations_vdb
  向量来源：edge 的 keywords + description
```

### 1.4 chunk schema 扩展

```
现状字段：tokens / content / full_doc_id / chunk_order_index

改造后新增：
  corpus_id   ← 语料库 ID
  source_file ← 源文件 ID
  ftype       ← 源文件格式
  chapter     ← 章节路径（支持嵌套如 1.2.3）
  domain      ← 所属领域（list[str]，允许多个）
```

## 二、domain 字段设计决策

### 2.1 决策：所有 C 和 KF 的 domain 字段为 list[str]

```
Concept.domain: list[str]
KnowledgeFragment.domain: list[str]
```

允许一个实体属于多个 domain，查询时 list 求交集。

**例**：

```
c_函数：
  name: "函数"
  domain: ["数学", "计算机科学"]

kf_001：
  c_id: c_函数
  content: "函数的定义：对于每一个 x..."
  domain: ["数学"]

kf_002：
  c_id: c_排序算法
  content: "快速排序是一种分治算法..."
  domain: ["计算机科学", "算法"]
```

### 2.2 domain 一致性规则

```
KF.domain ⊇ KF.c_id.domain 的超集

即：KF 可以比它的 C 拥有更多的 domain，
但至少要包含 C 的所有 domain（KF 不能比 C 的 domain 少）。
```

---

## 三、实体去重规则

### 3.1 决策：Concept 和 KnowledgeFragment 只按 exact name 合并

```
Concept 合并条件：name 完全相同（精确字符串匹配）
KnowledgeFragment 合并条件：name 完全相同（精确字符串匹配）
```

不设相似度阈值，不做模糊匹配。

**理由**：
- 相似但不相同的概念应该独立存在（如「矩阵的秩」vs「矩阵的行列式」）
- 合并权值交给后续的 weight 聚合机制，不依赖相似度

---

## 四、chapter 嵌套结构

### 4.1 设计

chapter 字段存储为字符串，格式为点分层级：

```
"1"           ← 第1章
"1.2"         ← 第1章第2节
"1.2.3"       ← 第1章第2节第3小节
```

### 4.2 解析函数

在 `utils.py` 新增解析函数：

```
parse_chapter(chapter_str) → dict
  输入："1.2.3"
  输出：{
    "chapter_level": 3,
    "parent": "1.2",
    "root": "1",
    "full_path": ["1", "2", "3"]
  }
```

### 4.3 查询过滤

```
查询"第2节下的所有内容"：
  → chapter 以 "*.2.*" 开头，或精确为 "2"
```

## 五、章节分段：按 chapter 而非 token_size 分块

### 5.1 改造逻辑

**现状**：完全按 token_size 分块，不考虑章节结构

**改造后**：优先按 chapter 分块，chapter 内部再按 token_size 切分

```
流程：
  1. 读取源文档的目录结构（LLM 推断或用户输入）
  2. 每个目录节点 = 一个 domain
  3. 每个目录节点下的内容 = 一个 KF 集合的候选
  4. 每个 KF 候选内部如果超过 max_token_size，再切分
```

**例**：

```
源文档目录：
├── 1.1 函数的概念
│   ├── 定义
│   ├── 例题
│   └── 习题
└── 1.2 函数的性质
    ├── 单调性
    └── 奇偶性

分块结果：
  KF1: domain="数学.函数.概念.定义"      content="..."
  KF2: domain="数学.函数.概念.例题"      content="..."
  KF3: domain="数学.函数.概念.习题"      content="..."
  KF4: domain="数学.函数.性质.单调性"    content="..."
  KF5: domain="数学.函数.性质.奇偶性"    content="..."
```

### 5.2 实现位置

在 `operate.py` 的 `chunking_by_token_size` 之前，增加一个 `chunking_by_chapter` 步骤。

## 六、extract_entities 改造

### 6.1 现状逻辑（hypergraphrag 原版）

```
输入：chunk（文本片段）
输出：("hyper-relation", <描述>, <评分>) + ("entity", <名称>, <类型>, <描述>, <评分>)
```

LLM 输出格式：扁平的二元组列表，entity 和 hyper-relation 交替出现，通过 `now_hyper_relation` 变量维护上下文归属。

### 6.2 改造后逻辑

拆分为两个独立的提取步骤：

**步骤1：提取 L1 Concept + 关系**

```
输入：chunk.content
输出：
  - Concept 列表：{ id, name, domain, description }
  - 关系列表：{ type, src, tgt, description }
           type ∈ { Depends, Related, Contains }
```

**步骤2：将 chunk 映射到 L1 Concept，生成 L2 KF**

```
输入：chunk 内容 + 分块后的 chunk 元信息（chapter, ftype, corpus_id, source_file）
输出：
  - KF 列表：{ id, c_id, content, ftype, corpus_id, chapter, source_file, domain }
```

### 6.3 关系类型限定

LLM 不再自由输出关系类型，而是从以下四种中选择：

```
Depends      ← 前置依赖（L1 内部）
Related      ← 关联（L1 内部）
Contains     ← 包含（L1→L2 跨空间）
Prerequisite ← 前置目标（L3→L3 或 L3→L1，仅当启用 L3 时）
```

## 七、query 模式扩展

### 7.1 现状 query 模式

```
local  → 通过 entities_vdb 找相关实体，再找关联的 chunk
global → 通过 hyperedges_vdb 找相关超边，再找关联的实体和 chunk
hybrid → local + global 合并
naive  → 纯向量检索
```

### 7.2 改造后 query 模式

```
local         ← 查具体片段内容（L2 KF + L1 C）
global        ← 查概念关联网络（L1 C 之间的关系）
hybrid        ← local + global
goal_driven   ← 从 L3 Goal 反向查 L1 C + L2 KF（仅当启用 L3）
```

### 7.3 domain 过滤

所有 query 模式均支持 `domain` 参数过滤：

```
query(domains=["数学", "函数"], mode="local")
→ 只返回 domain 包含"数学"或"函数"的 KF 和 C
```

## 十、关键设计决策汇总

| 编号 | 决策 | 结论 |
|------|------|------|
| D1 | domain 字段类型 | list[str]，允许多个 |
| D2 | Concept 合并规则 | exact name 精确匹配，不设相似度阈值 |
| D3 | KF 合并规则 | 同上 |
| D4 | chapter 存储格式 | 字符串点分："1.2.3" |
| D5 | 分块策略 | 先按 chapter 分块，chapter 内部再按 token_size 切分 |
| D6 | L3 是否必须 | 不必须，goal_driven 模式可按需启用 |
| D7 | hyperedge 存储 | 不独立存储超边，通过普通边（Contains / Depends 等）实现 |
| D8 | domain 一致性 | KF.domain ⊇ KF.c_id.domain |

**两个分支**：

**分支 A（chapter_hint 有值）**：按章节结构切分

```
  4. 每个子 chunk 继承该章节的 chapter 路径
```

**分支 B（无 chapter_hint）**：回退到纯 token 分块 + 正则推断 chapter
```
逻辑：
  1. 调用 _sub_chunking_by_token 全量分块
  2. 对每个 chunk 内容用正则推断 chapter（/第[一二三四五六七八九十\\d]+章/ 等）
  3. 推断失败则 chapter 为 ""
```

**输出格式**：与 `TextChunkSchema` 对应（tokens / content / chunk_order_index / corpus_id / source_file / ftype / chapter / domain）
